Buttons

Action elements that guide buyers through discovery, quoting, and ordering flows in B2B seafood marketplaces.

What Are Buttons?

Buttons are interactive elements that trigger actions when clicked or tapped. They guide users through critical workflows like placing orders, requesting quotes, confirming transactions, or navigating to key destinations in the seafood marketplace. Buttons should have clear, action-oriented labels and provide immediate visual feedback on interaction.

When to Use Buttons

Use buttons when:

  • ✅ Triggering an immediate action (submit form, add to cart, confirm order)
  • ✅ Initiating a system process (generate quote, export data, send email)
  • ✅ Navigating to a critical destination (view cart, proceed to checkout)
  • ✅ Opening important modals or dialogs (edit profile, delete item)
  • ✅ The action is the primary focus of the page or section

When NOT to Use Buttons

Avoid using buttons when:

  • ❌ Simple navigation between pages (use text links instead)
  • ❌ Toggling UI states like show/hide (use toggle switches)
  • ❌ Selecting from multiple options (use radio buttons or checkboxes)
  • ❌ Sorting or filtering data (use dropdown menus or filter chips)
  • ❌ Opening external links (use text links with external icon)

Button Sizes

Buttons come in five sizes to match different contexts: xsmall (28px), small (32px), medium (40px - default), large (48px), and xlarge (56px). Use smaller sizes for compact interfaces and larger sizes for prominent calls-to-action.

Button Variants

Different visual styles for different hierarchy levels. Use primary for main actions, secondary for alternative options, outline for tertiary actions, and ghost for subtle interactions.

<!-- Primary Button - Main Actions --> <button style=" background: #025E96; color: #FFFFFF; padding: 0 24px; height: 40px; border: none; border-radius: 8px; font-family: 'DM Sans', sans-serif; font-size: 16px; font-weight: 600; cursor: pointer; "> Primary </button> <!-- Secondary Button - Alternative Actions --> <button style=" background: #F7FAFC; color: #025E96; padding: 0 24px; height: 40px; border: 1px solid #025E96; border-radius: 8px; font-family: 'DM Sans', sans-serif; font-size: 16px; font-weight: 600; cursor: pointer; "> Secondary </button> <!-- Outline Button - Tertiary Actions --> <button style=" background: transparent; color: #025E96; padding: 0 24px; height: 40px; border: 1px solid #CBD5E0; border-radius: 8px; font-family: 'DM Sans', sans-serif; font-size: 16px; font-weight: 600; cursor: pointer; "> Outline </button> <!-- Ghost Button - Subtle Actions --> <button style=" background: transparent; color: #025E96; padding: 0 24px; height: 40px; border: none; border-radius: 8px; font-family: 'DM Sans', sans-serif; font-size: 16px; font-weight: 600; cursor: pointer; "> Ghost </button>

Button States

Buttons display different visual states to provide feedback: default, hover, active (pressed), disabled, and loading.

Best Practices

✅ Do

  • Use clear, action-oriented labels ("Add to Cart", "Request Quote", "Submit Order")
  • Keep labels concise (2-4 words maximum)
  • Use only ONE primary button per section to maintain hierarchy
  • Provide immediate visual feedback on hover and click
  • Ensure minimum 44x44px touch targets for mobile accessibility
  • Maintain consistent button sizing within the same context
  • Use sentence case for button labels ("Add to cart" not "ADD TO CART")
  • Disable buttons during processing to prevent double-submission

❌ Don't

  • Don't use vague labels like "Click here", "OK", or "Submit" without context
  • Don't make buttons too small (minimum 32px height, 44px for touch)
  • Don't use multiple primary buttons competing for attention
  • Don't use all caps text (reduces readability and feels aggressive)
  • Don't disable buttons without explaining why they're disabled
  • Don't use buttons for simple navigation (use links instead)
  • Don't place buttons too close together (minimum 8px spacing)
  • Don't use colors that fail WCAG contrast requirements

♿ Accessibility Guidelines

  • ✅ Minimum size: 44x44px touch target (WCAG 2.5.5 Level AAA)
  • ✅ Color contrast: 4.5:1 minimum for text, 3:1 for non-text (Waves-500 meets WCAG AA)
  • ✅ Keyboard navigation: Focusable via Tab key, activatable with Enter or Space
  • ✅ Focus indicator: Visible 3px ring on focus (use outline or box-shadow)
  • ✅ Screen readers: Use descriptive labels; add aria-label if visual label is unclear
  • ✅ Disabled state: Use disabled attribute and aria-disabled="true"
  • ✅ Loading state: Add aria-busy="true" and descriptive aria-label ("Processing...")
  • ✅ Icon-only buttons: Always include aria-label describing the action

Related Components

  • Input Fields - Often paired with submit buttons
  • Badges - For status indicators, not actions
  • Chips - For removable tags and filters
